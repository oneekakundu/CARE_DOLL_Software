                 CARE DOLL: ARCHITECTURE & TECHNICAL RATIONALE
========================================================================

1. COMPLETE SYSTEM ARCHITECTURE
--------------------------------
CARE DOLL is an offline-first AI emergency assistant designed primarily for Electric Vehicle (EV) accident assistance and safety guidance. The entire system operates locally without relying on external cloud APIs or network connectivity.

The end-to-end request-response pipeline is structured as follows:

   [Microphone / Audio Input]
               │
               ▼
       [Voice_Input Module] (Records user audio input)
               │
               ▼
    [Faster-Whisper STT] (Transcribes audio WAV to text query)
               │
               ▼
       [Text Query Input]
               │
               ▼
     [RAG Retrieval Pipeline] (Queries pre-indexed documents)
               │
               ▼
     [FAISS Vector Store] (Performs dense vector similarity search)
               │
               ▼
 [BGE-small-en-v1.5 Embeddings] (Encodes queries & document chunks)
               │
               ▼
       [Ollama Local Server] (Orchestrates LLM execution)
               │
               ▼
      [Qwen 2.5:3b LLM] (Generates grounded response from context)
               │
               ▼
    [Grounded Text Response]
               │
               ▼
     [Voice_Output Pipeline] (Routes request via speak() API)
               │
               ▼
       [EngineSelector] (Evaluates mode & profile readiness)
            ┌──┴──┐
            ▼     ▼
     [Piper TTS] [XTTS v2]
            └──┬──┘
               ▼
       [WAV Audio Output] (Saves synthesized audio to data/tts/)
               │
               ▼
  [AudioPlayer (sounddevice)] (Streams WAV output to system speakers)

2. MODEL AND TECHNOLOGY RATIONALE
------------------------------------------------------------------------

A. Speech-to-Text: Faster-Whisper (Whisper Base)
- Role: Converts spoken audio input into structured text queries.
- Technical Rationale: Faster-Whisper utilizes CTranslate2 for CPU-optimized inference with int8 quantization. It delivers accurate, low-latency transcription locally without requiring internet access or high GPU VRAM.

B. Vector Similarity Search: FAISS
- Role: Stores embedding vectors of document chunks and executes high-speed similarity searches.
- Technical Rationale: FAISS (Facebook AI Similarity Search) provides fast, memory-efficient vector searching using Inner Product (IP) distance metrics. It operates completely in-process, requiring no external database servers or background daemon processes.

C. Text Embeddings: BAAI/bge-small-en-v1.5
- Role: Converts raw text document chunks and user queries into 384-dimensional dense vector embeddings.
- Technical Rationale: BAAI/bge-small-en-v1.5 offers top-tier retrieval performance among lightweight embedding models. It loads in under 1.5 seconds and runs efficiently on CPU with a minimal memory footprint.

D. Local LLM Orchestration: Ollama + Qwen 2.5:3b
- Role: Generates clear, structured response text grounded strictly in retrieved safety manuals.
- Technical Rationale: Qwen 2.5 (3B parameters) executed via Ollama balances high instruction-following precision with low resource consumption. Running via Ollama ensures local execution. The prompt conditions the LLM to generate responses grounded in retrieved context; however, standard AI disclaimers apply as no LLM is guaranteed to be completely hallucination-free.

E. Default Emergency TTS: Piper (`en_US-lessac-medium.onnx`)
- Role: Renders rapid, lightweight, and reliable voice audio for emergency responses and safety alerts.
- Technical Rationale: Piper is a C++ compiled ONNX runtime engine optimized for low-resource devices. It synthesizes audio with minimal latency, requires zero heavy deep-learning dependencies, and guarantees voice response availability during critical emergencies.

F. Personalized Companion TTS: XTTS v2
- Role: Provides optional personalized voice synthesis conditioned on a short reference recording.
- Technical Rationale: XTTS v2 enables zero-shot voice cloning for non-emergency companion interactions. It remains completely optional so as not to compromise emergency system startup or hardware reliability.

3. RAG EXTRACTION AND INGESTION PROCEDURE
------------------------------------------------------------------------
To ensure query-time operation remains instantaneous, CARE DOLL decouples document extraction from real-time query execution:

   documents/raw/ (Raw PDF manuals)
          │
          ▼
   [Docling Extraction] (Run via offline extraction script)
          │
          ▼
   documents/extracted/ (Structured Markdown & JSON files)
          │
          ▼
   [FAISS Vector Store Indexing] (Embedded via BGE-small)

During normal application startup (`main.py`), the system loads pre-extracted content from `documents/extracted/` and pre-built FAISS indices. This design eliminates runtime PDF parsing and heavy OCR model initialization, enabling instant startup.

------------------------------------------------------------------------
4. LAZY LOADING AND RESOURCE MANAGEMENT
------------------------------------------------------------------------
To preserve system RAM and VRAM during critical operations, XTTS v2 uses lazy loading and lazy imports.

Key Resource Rules:
- XTTS v2 is not loaded during normal CARE DOLL startup, RAG indexing, or LLM response generation.
- Piper operates independently and is immediately available upon system boot.
- Heavy PyTorch and deep-learning packages for XTTS v2 are imported lazily only when a valid companion speech request is executed.
- Hardware execution (GPU CUDA vs CPU) is determined dynamically before model initialization.

Logical Resource & Execution Flow:

CARE DOLL Startup
       │
       ▼
  Piper Available
       │
       ▼
 Normal Operation
       │
       ├── Emergency Request ───────────────► Piper
       │
       └── Companion Request
               │
               ▼
      Personalized Profile Ready?
         ┌─────┴─────┐
        No          Yes
         │           │
         ▼           ▼
       Piper      XTTS v2
                     │
                     ▼
             Lazy Model Loading

------------------------------------------------------------------------
5. DYNAMIC DEVICE DETECTION
------------------------------------------------------------------------
Before initializing the XTTS v2 engine, CARE DOLL dynamically inspects available hardware using PyTorch:

```python
torch.cuda.is_available()
```

- If CUDA is available: XTTS v2 is configured for GPU execution (CUDA), providing faster audio synthesis.
- If CUDA is unavailable: XTTS v2 gracefully falls back to CPU execution.

The system dynamically detects device availability at runtime and never assumes CUDA hardware is present.

------------------------------------------------------------------------
6. VOICE PROFILE VALIDATION AND CONDITIONING
------------------------------------------------------------------------
Voice profiles are stored in structured directories:

data/voices/profiles/primary_user/
├── profile.json
└── reference.wav

Profile metadata (`profile.json`) defines profile configuration:
{
  "profile_name": "primary_user",
  "engine": "xtts_v2",
  "language": "en",
  "reference_audio": "reference.wav",
  "enabled": true,
  "consent_confirmed": true
}

Validation Requirements:
A profile is considered READY only when all 7 validation rules pass:
1. Profile directory exists.
2. `profile.json` exists.
3. `profile.json` contains valid JSON.
4. Reference audio file (`reference.wav`) exists.
5. Reference audio file is readable and non-empty (>0 bytes).
6. `consent_confirmed` is `true`.
7. `enabled` is `true`.

Speaker Conditioning Rationale:
XTTS v2 does not train a new neural voice model from scratch. Instead, it extracts acoustic speaker characteristics from `reference.wav` to condition zero-shot audio synthesis. Strict consent (`consent_confirmed: true`) and explicit user enablement (`enabled: true`) ensure privacy compliance and user authorization before conditioning takes place.

------------------------------------------------------------------------
7. EMERGENCY RELIABILITY DESIGN
------------------------------------------------------------------------
Emergency assistance reliability is the primary design mandate of CARE DOLL:

   Emergency Functionality ──► Must NOT Depend On ──► XTTS v2

Fallback Architecture:
If XTTS v2 is uninstalled, dependency conflicts occur, profile validation fails, or audio hardware is restricted, CARE DOLL enforces an automatic fallback chain:

   Personalized XTTS v2 Unavailable / Profile Not Ready
                          │
                          ▼
                    Piper Fallback
                          │
                          ▼
       Emergency Assistant Continues Operating Seamlessly

------------------------------------------------------------------------
8. FUTURE IMPROVEMENTS
------------------------------------------------------------------------
The following capabilities represent planned future work:

1. Multilingual Support: Integration of multilingual Piper ONNX models and Whisper transcription weights for non-English emergency response.
2. Dynamic Voice Profiles: Context-aware vocal inflection (e.g., rapid, high-pitch alerts for active crashes vs. calm, deliberate tone for maintenance guides).
3. Wake-Word Integration: Continuous hands-free monitoring utilizing OpenWakeWord for voice activation in vehicle cabins.
4. Hardware Portability: Target cross-compilation for embedded single-board computers (e.g., Raspberry Pi or automotive ARM hardware).

Note: These features are future design goals and are not presented as current implementations.
