========================================================================
                      CARE DOLL: SYSTEM & MODEL EXPLANATION
========================================================================

1. ARCHITECTURAL OVERVIEW & WORKFLOW
------------------------------------
CARE DOLL is an offline-first AI Emergency Assistant designed for Electric-Vehicle 
accidents and first-aid reference. It coordinates speech input, transcription, 
retrieval-augmented generation (RAG), and modular text-to-speech (TTS) 
technologies to work completely locally and offline.

The end-to-end request-response pipeline is structured as follows:

   [User Input (Microphone / WAV)]
                 |
                 v
       [Voice_Input Module] (Records user speech)
                 |
                 v
     [Speech-to-Text (Whisper)] (Transcribes audio into text query)
                 |
                 v
      [RAG Ingestion / Ingested Markdown] (Reads pre-extracted Markdown from documents/extracted/)
                 |
                 v
   [Retriever (FAISS + BGE-small)] (Identifies top-K most semantically relevant chunks)
                 |
                 v
       [Ollama (Qwen 2.5:3b)] (Generates a context-grounded response using the chunks)
                 |
                 v
       [Voice_Output Pipeline] (Validates text structure and starts audio rendering)
                 |
                 v
      [Piper TTS (piper.exe)] (Synthesizes text into a natural-sounding voice WAV)
                 |
                 v
   [data/tts/response_*.wav] (Saves the output WAV file)
                 |
                 v
      [AudioPlayer (sounddevice)] (Streams the output to system speakers)


2. COMPONENT & MODEL ROLES (WHY THESE MODELS?)
----------------------------------------------

A. Text-to-Speech: Piper TTS (en_US-lessac-medium.onnx)
- Role: Synthesizes high-quality, local, offline voice responses from text.
- Why: Piper is written in C++ and optimized for low-resource environments (like Raspberry Pi 
  or older laptops). It runs with near-zero latency, does not require an active internet connection, 
  and sounds significantly more natural and human-like than legacy robotic Windows SAPI voices.
- Execution: Managed by `Voice_Output/`. It runs a validation check on startup to verify 
  executables, model files, and output file writing.

B. Large Language Model: Ollama (Qwen 2.5:3b)
- Role: Generates responses grounded strictly within the retrieved documentation.
- Why: Qwen 2.5 (3B) is a highly capable small language model, demonstrating superior 
  instruction-following and factual parsing capabilities for its size. It fits within standard 
  consumer memory limits, ensuring the system remains offline-first.
- Grounding: A strict custom prompt prevents the model from hallucinating outside the retrieved RAG manual content.

C. Vector Store: FAISS (Facebook AI Similarity Search)
- Role: Indexes text chunks and performs high-speed similarity search using Inner Product (IP) metric.
- Why: Extremely fast, lightweight, and requires no running database servers.

D. Embedding Model: Sentence-Transformers (BAAI/bge-small-en-v1.5)
- Role: Translates raw text chunks and user queries into 384-dimensional dense vectors.
- Why: BGE-small is one of the highest-rated lightweight embeddings models, allowing rapid 
  quantization and loading (under 1.5s) on consumer CPUs with minimal memory footprints.

E. Speech-to-Text: Faster-Whisper (Whisper Base)
- Role: Transcribes recorded WAV files into text.
- Why: Employs CTranslate2 translation engines to perform Whisper inference on CPUs with `int8` 
  quantization, rendering transcrips with high accuracy and speed.


3. THE RAG EXTRACTION AND INGESTION PROCEDURE
---------------------------------------------
To ensure query startups are instantaneous, the extraction pipeline is completely decoupled:

1. Target Pathing:
   - Raw files (PDFs) are stored in `documents/raw/`.
   - Extracted files are stored in `documents/extracted/`.
2. Document Extraction:
   - Run `python -m RAG.extract_docs` to parse raw PDF files using Docling.
   - Outputs markdown (`.md`) and JSON files directly into `documents/extracted/`.
3. Ingestion:
   - When running `main.py`, the RAG pipeline scans only `documents/extracted/`.
   - Since Markdown is lightweight, building or loading the FAISS vector index takes <0.1 seconds, 
     entirely bypassing PyTorch OCR model initialization delays on startup.


4. FUTURE IMPROVEMENTS
----------------------
1. Multilingual Support: Load alternative Piper languages and Whisper transcription weights.
2. Dynamic Voice Profiles: Adapt the speech profile style dynamically (e.g. fast, high-pitch 
   "emergency" alerts during crashes; calm, low-pace "companion" voices for instruction guides).
3. Wake-Word Integration: Incorporate OpenWakeWord for continuous, hands-free active monitoring.
4. Portability: Compile the program dependencies to deploy directly on physical companion hardware (Raspberry Pi/Linux boards).
